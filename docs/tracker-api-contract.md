# Контракт API трекера (прогон 4: доска кухни)

Фиксируется **до** реализации, чтобы бэкенд и фронт собирались параллельно.
Префикс REST: `/api/tracker`. Реальное время: `WS /ws/tracker/{point_code}`.

## Аутентификация и скоуп

JWT персонала — тот же, что у CMS (`POST /api/staff/auth/login`).
`Authorization: Bearer <jwt>`; в деве плюс `X-Hotel-Subdomain: crystal`.

**Три независимые проверки на каждом запросе и на подключении к WS:**

1. токен валиден и не протух;
2. отель токена совпадает с отелем запроса (тенант выбран поддоменом);
3. **сотрудник привязан к этой точке исполнения** (`StaffAssignment`).

Третья — не формальность: повар с кухни не должен видеть заявки SPA, а
сотрудник соседнего отеля — вообще ничего. Отказ: `403 point_not_assigned`.

> **Урок прошлого прогона.** У WebSocket **нет HTTP-middleware**: ни
> аутентификации, ни резолвера тенанта, ни языка. Всё перечисленное WS-хендлер
> обязан делать явно и сам. Полагаться на то, что «мидлварь разберётся», —
> прямой путь к открытому наружу каналу.

## 1. Точки исполнения сотрудника

### `GET /api/tracker/points`

```jsonc
{
  "points": [
    {"id": "...", "code": "kitchen", "title": "Кухня ресторана",
     "kind": "kitchen", "level": "lead", "sla_minutes": 20,
     "active_count": 3, "new_count": 1}
  ]
}
```

Только те точки, к которым сотрудник привязан. Пустой список — не ошибка:
сотрудника просто ещё не назначили, UI показывает это отдельным экраном.

## 2. Доска

### `GET /api/tracker/orders?point=kitchen&scope=active`

`scope`: `active` (по умолчанию) — всё, что не в терминальном статусе;
`history` — терминальные, за последние сутки, новыми сверху.

```jsonc
{
  "point": {"id": "...", "code": "kitchen", "title": "Кухня ресторана"},
  "scope": "active",
  "server_time": "2026-07-21T19:05:00+03:00",
  "columns": [
    // color_token — имя токена темы, а не цвет: палитра живёт только в токенах
    {"code": "new",       "title": "Новый",    "color_token": "info",    "orders": [ /* §3 */ ]},
    {"code": "accepted",  "title": "Принят",   "color_token": "info",    "orders": []},
    {"code": "preparing", "title": "Готовится","color_token": "warning", "orders": []},
    {"code": "on_the_way","title": "В пути",   "color_token": "warning", "orders": []}
  ]
}
```

Колонки строятся из **пресета статусов отеля**, а не захардкожены на клиенте:
у ресторана «готовится → в пути», у SPA будет своё. Терминальные статусы в
`active` не попадают.

## 3. Объект заказа на доске

Надстройка над гостевым объектом (`docs/guest-api-contract.md` §6) — те же
поля плюс то, что нужно исполнителю:

```jsonc
{
  // ...всё из гостевого объекта: id, number, type, status, status_flow,
  // history, room, location, delivery_mode, requested_time, comment, total,
  // items, field_values...
  "execution_point": {"id": "...", "code": "kitchen", "title": "Кухня ресторана"},
  "assignee": {"id": "...", "name": "Пётр, повар"} | null,
  "accepted_at": "2026-07-21T19:06:00+03:00" | null,
  "waiting_minutes": 4,          // сколько заказ уже ждёт — для сортировки и SLA
  "is_overdue": false,           // ждёт дольше порога точки
  "next_statuses": [             // куда можно двинуть ИЗ текущего статуса
    {"code": "preparing", "title": "Готовится"}
  ],
  "can_cancel": true
}
```

`next_statuses` считает сервер: клиент не должен знать правила переходов, а
кнопки на карточке обязаны совпадать с тем, что сервер реально примет.

**Заявка-услуга приезжает на доску тем же объектом.** Отличается только тело
карточки: у еды — список позиций, у заявки — ответы на поля (`field_values`).
Доска, действия, статусы и WebSocket общие; отдельного «трекера услуг» нет и
не должно появиться (см. [`offering-types.md`](offering-types.md)).

Заявки маршрутизируются в свои отделы: такси — на консьержа, уборка — в
хозслужбу. Доска кухни их не показывает, и наоборот — это обычная работа
`Route`, а не новая логика.

## 4. Один заказ и действия над ним

Все требуют привязки к точке заказа (см. выше).

| Метод | Путь | Что делает |
|---|---|---|
| GET | `/api/tracker/order/{id}` | заказ отдельно от доски |
| POST | `/api/tracker/order/{id}/accept` | берёт заказ в работу |
| POST | `/api/tracker/order/{id}/status` | двигает статус |
| POST | `/api/tracker/order/{id}/cancel` | отменяет |

`GET` нужен для прямой ссылки на заказ (`/tracker/order/{id}` в UI, ссылка из
уведомления). Брать заказ из снимка доски для этого недостаточно: открытая
ссылка может вести на заказ, которого на текущей вкладке нет — например, уже
завершённый, когда открыта «В работе».

### `accept`
```jsonc
{}   // тело не нужно
```
Переводит заказ в первый неначальный статус пресета (в демо — «Принят») и
проставляет `assignee` = текущий сотрудник, `accepted_at` = сейчас.
Повторный вызов на уже принятом заказе — `409 already_accepted` с текущим
исполнителем в теле: два официанта не должны «перехватывать» заявку молча.

### `status`
```jsonc
{"status": "preparing", "comment": ""}
```
`422 invalid_transition`, если целевого статуса нет в `next_statuses`.
Если исполнитель ещё не назначен — назначается тот, кто двинул статус.

### `cancel`
```jsonc
{"reason": "нет продуктов"}
```
`409 cancel_not_allowed`, если заказ уже терминальный.

Ответ всех трёх — обновлённый объект заказа (§3).

Каждый переход эмитит событие **после коммита**, поэтому изменение
одновременно прилетает и гостю (`/ws/guest/order/{id}`), и на все остальные
устройства трекера.

## 5. WebSocket

### `WS /ws/tracker/{point_code}?token=<jwt>&hotel=<subdomain>&lang=ru`

`hotel` и `lang` — параметрами, потому что у WS нет ни заголовков от
vite-прокси, ни Accept-Language. В проде отель берётся из Host.

**Реконсиляция, как и у гостя.** Сервер шлёт полный снимок доски сразу после
подключения и после каждого события точки. Клиент состояние заменяет, а не
докручивает: переподключение после потери сети лечит само себя, а гонка
WS с REST становится невозможной.

```jsonc
// при подключении и на каждое событие
{
  "type": "tracker.snapshot",
  "event": "order.created",       // "connected" у первого снимка
  "order_id": "...",              // какой заказ дёрнулся — для звука и подсветки
  "board": { /* тело GET /api/tracker/orders, §2 */ }
}

{"type": "ping"} → {"type": "pong"}
```

Полный снимок доски на каждое событие — осознанный размен: на кухне
одновременно живут единицы заказов, и простота инварианта важнее экономии
трафика. Если точка вырастет до сотен активных заявок, здесь появится
дельта-режим.

Коды закрытия: `4401` — токен не подошёл, `4403` — точка недоступна,
`4404` — отель не определён.

`4403` намеренно один и тот же и для «точки не существует», и для «не твоя
точка»: по разнице кодов чужой мог бы перебором узнать, какие точки есть в
отеле. У REST такого ограничения нет — там сотрудник уже прошёл авторизацию,
и `404` на несуществующую точку помогает отладке.

## 6. Маршрутизация заказа к точке

Точка исполнения резолвится при создании заказа и дальше не пересчитывается —
перенастройка маршрутов не должна переносить вчерашние заказы на другую кухню.

Порядок разрешения:

1. явный `Route` (категория → точка) с наименьшим `priority`;
2. точка с тем же `code`, что у категории (соглашение «категория = точка»);
3. если в отеле ровно одна активная точка — она;
4. иначе `422 no_route`.

Пункты 2–3 нужны, чтобы категория, только что созданная в CMS, не ломала
гостю оформление заказа: маршрут для неё ещё не настроен, а заказать её уже
можно.
