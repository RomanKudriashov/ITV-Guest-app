# Типы предложений: как система остаётся одной, а не тремя

Документ описывает, что именно отличается между типами услуг и — важнее — что
**не** отличается. Он же служит правилом для будущих типов (`info`,
`slot_booking`): если новый тип требует форка `Order`, гостевого потока или
трекера — треснула модель, а не тип.

## Принцип

Один заказ, один трекер, один гостевой поток. Тип влияет ровно на три вещи:

1. **чем гость наполняет заявку** — корзина позиций или форма полей;
2. **что показывать в теле карточки** — список позиций или ответы на поля;
3. **какие проверки применить** при отправке.

Всё остальное — маршрутизация в отдел, статусы, события, WebSocket,
идемпотентность, история, изоляция тенантов — общее и типа не знает.

## Реестр поведений

Различия собраны в ОДНУ таблицу (`apps/catalog/offerings.py`), а не разбросаны
условиями по коду. Прикладной код спрашивает у реестра флаг, а не сравнивает
строку типа:

```python
behaviour = behaviour_for(item.type)
if behaviour.uses_fields:   # а НЕ: if item.type == "service_request"
    ...
```

| Поведение | `product` (еда) | `service_request` (заявка) | `info` (инфо) | `slot` (бронь) |
|---|---|---|---|---|
| `creates_order` | да | да | **нет** | да |
| `order_type` | `cart` | `request` | — | `booking` |
| `allows_multiple_lines` | да | нет | — | нет |
| `uses_modifiers` | да | нет | нет | нет |
| `uses_fields` | нет | да | нет | нет |
| `uses_content` | нет | нет | **да** | нет |
| `uses_slots` | нет | нет | нет | **да** |
| `default_location_mode` | `delivery` | `room` | `none` | `none` |
| `priced` | всегда | необязательно | никогда | необязательно |

`info` — единственный тип, который **не создаёт `Order`**: это страница только
для чтения (об отеле, Wi-Fi, правила). Он не маршрутизируется и не виден
трекеру. `creates_order=false` — новый флаг, ради которого гостевой поток
получил ровно одну развилку: «эту позицию нельзя заказать».

`slot` создаёт обычный `Order` типа `booking` со ссылкой на забронированный
слот (`SlotBooking`). Дальше — тот же поток: маршрут в отдел, живой статус,
история, отмена (освобождает слот). Тело карточки на доске показывает слот
вместо позиций — та же развилка «по данным», что и `field_values` у заявки.

Разница между «есть цена» и «цены нет» — не тип, а **свойство позиции**:
у такси цена может быть, у уборки её нет. Поэтому `Item.price` стал nullable,
а не «услуги всегда бесплатны».

## Локация

Раньше витрина всегда спрашивала «куда доставить». Для такси это бессмысленно
(точка подачи — поле заявки), для уборки избыточно (номер и так известен).
Поэтому у позиции есть `location_mode`:

| Значение | Что делает витрина | Пример |
|---|---|---|
| `delivery` | спрашивает локацию и уточнение | еда |
| `room` | молча берёт номер гостя | уборка |
| `none` | не спрашивает и не заполняет | такси (адрес — поле) |

Значение по умолчанию берётся из реестра поведений при создании позиции, но
отель может переопределить его в CMS. Сервер валидирует по тому же полю: где
`location_mode != delivery`, локация не требуется и не навязывается.

## Заказ

`Order` один. Заявка-услуга — это заказ с **одной** позицией (сама услуга) и
заполненным `field_values`:

```jsonc
{
  "type": "request",
  "items": [{"title": "Такси", "quantity": 1, "unit_price": null, ...}],
  "field_values": [
    {"code": "destination", "label": "Куда", "field_type": "text",
     "value": "Аэропорт Пулково", "display": "Аэропорт Пулково"},
    {"code": "passengers", "label": "Сколько человек", "field_type": "count",
     "value": 3, "display": "3"}
  ]
}
```

Позиция у заявки есть намеренно, хотя «корзины» нет. Это то, что позволяет
маршрутизации, ценообразованию, истории и снапшотам работать одним и тем же
кодом: маршрут резолвится по категории позиции, стоп-лист и расписание
проверяются той же функцией доступности, название фиксируется тем же
снапшотом. Убрать позицию — значит завести второй путь для всего перечисленного.

`field_values` — **снимок**, как и цены: заявка обязана пережить
переименование и удаление полей в CMS.

## Поля заявки

`RequestField` — на позицию, по образцу `ModifierGroup`:

| Поле | Смысл |
|---|---|
| `code` | стабильный ключ ответа |
| `label{lang}` | подпись |
| `field_type` | `text` · `number` · `count` · `date` · `time` · `select` |
| `is_required` | обязательность (проверяет сервер) |
| `options` | варианты для `select`: `[{"value","label":{lang}}]` |
| `min_value` / `max_value` | границы для `number` и `count` |
| `help_text{lang}` | подсказка под полем |
| `sort_order` | порядок в форме |

Разбор и проверка значения — таблица по `field_type`
(`apps/catalog/request_fields.py`). Это единственное место в проекте с
разветвлением по типу, и оно про **тип поля**, а не про тип услуги.

## Где ветвление осталось — и почему

Полный список мест, где код знает о различиях типов:

1. `apps/catalog/offerings.py` — сам реестр (это и есть его работа);
2. `apps/catalog/request_fields.py` — разбор значения по типу ПОЛЯ;
3. валидация заказа в `orders/services.py` — проверки по флагам реестра
   (`creates_order`, `uses_modifiers`, `uses_fields`, `uses_slots`), без
   сравнения строк типа;
4. `apps/catalog/slots.py` — расчёт доступности и транзакционная бронь
   (изолированы в своём модуле, вызываются по флагу `uses_slots`);
5. фронт: тело карточки/страницы и тело карточки на доске — разная вёрстка
   одних и тех же данных, выбор по наличию блока (`content`/`slot`/
   `field_values`), а не по строке типа.

Маршрутизация, статусы, шина событий, WebSocket, идемпотентность, история,
трекер, изоляция — ни одной ветки по типу.
