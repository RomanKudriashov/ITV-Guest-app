# Контракт типов `info` и `slot`

Фиксируется **до** реализации. Два новых типа предложений ложатся в тот же
`Order`, гостевой поток и трекер — см. [`offering-types.md`](offering-types.md).

## Тип `info` — страница только для чтения

Оффер без заказа. `creates_order=false`: не создаёт `Order`, не
маршрутизируется, трекер его не видит.

### Каталог и карточка (гость)

`GET /api/v1/guest/catalog?type=info` — те же категории/позиции, что у остальных
типов. У позиции `info`:

```jsonc
{
  "id": "...", "code": "wifi", "type": "info",
  "title": "Wi-Fi и интернет",
  "description": "Как подключиться",       // краткое — для строки списка
  "images": ["http://.../card.webp"],
  "content": "## Сеть\nCrystal-Guest\n**Пароль:** 12345",  // тело, {lang}→строка
  "has_modifiers": false, "has_fields": false, "is_orderable": false,
  "is_available": true
}
```

`GET /api/v1/guest/item/{id}` у `info` возвращает то же + `content`. Кнопки заказа
нет: `is_orderable=false`. Попытка заказать `info` — `422 not_orderable`.

### CMS

`Item.content` — переводимое поле (`{lang}`), редактируется вместо
цены/модификаторов/полей. Markdown-подобный текст; рендер — на клиенте.

## Тип `slot` — бронь слота

Бронь ресурса (СПА-кабинет, тренер, зал): гость выбирает дату → видит свободные
слоты → бронирует → обычный `Order` типа `booking` со ссылкой на слот.

### Модель

`SlotConfig` — конфигурация бронируемой позиции (OneToOne к `Item` типа `slot`):

```jsonc
{
  "id": "...", "item_id": "...",
  "duration_minutes": 60,       // длина слота
  "capacity": 2,                // сколько гостей на один слот одновременно
  "schedule_id": "...",         // рабочие часы (переиспользуется Schedule)
  "execution_point_id": "...",  // отдел-исполнитель (напр. СПА)
  "lead_minutes": 0,            // за сколько минимум до начала можно бронировать
  "horizon_days": 14            // на сколько дней вперёд открыта бронь
}
```

`SlotBooking` — сама бронь: `item`, `order`, `starts_at`, `ends_at`,
`is_active`. Отмена заказа → `is_active=false` → слот освобождается.

### Доступность (гость)

`GET /api/v1/guest/slots?item_id=<id>&date=YYYY-MM-DD`

```jsonc
{
  "date": "2026-07-25",
  "duration_minutes": 60,
  "capacity": 2,
  "slots": [
    {"starts_at": "2026-07-25T10:00:00+03:00", "ends_at": "...11:00...",
     "capacity_left": 2, "available": true},
    {"starts_at": "...11:00...", "capacity_left": 0, "available": false}
  ]
}
```

Свободные слоты = рабочие часы расписания (в TZ отеля) нарезаны на интервалы
длиной `duration_minutes`, из вместимости вычтены активные брони на это время.
Слот `available` при `capacity_left > 0`, не в прошлом и в пределах горизонта.

### Бронь (гость)

`POST /api/v1/guest/order` тем же эндпоинтом:

```jsonc
{"lines": [{"item_id": "<slot item>"}], "slot_start": "2026-07-25T10:00:00+03:00"}
```

Отказы, помимо общих:

| Код | `code` | Когда |
|---|---|---|
| 422 | `slot_required` | у позиции `slot` не передан `slot_start` |
| 422 | `slot_not_offered` | `slot_start` не попадает в рабочие часы/шаг |
| 422 | `slot_in_past` | слот в прошлом или дальше горизонта |
| 409 | `slot_taken` | вместимость исчерпана (гонка проиграна) |

**Защита от двойной брони — критично.** Бронь идёт в транзакции с
`SELECT … FOR UPDATE` на строке `SlotConfig`: это сериализует одновременные
брони одного ресурса. Проверка вместимости внутри блокировки не может быть
обогнана — два гостя за последний слот: успевает один, второй получает
`slot_taken`.

### Объект заказа (§6 гостевого контракта) — блок `slot`

Непусто только у брони; клиент и трекер рисуют им тело карточки:

```jsonc
{
  // ...общий объект заказа...
  "type": "booking",
  "slot": {
    "resource_title": "СПА-кабинет",
    "starts_at": "2026-07-25T10:00:00+03:00",
    "ends_at": "2026-07-25T11:00:00+03:00",
    "duration_minutes": 60
  }
}
```

Отмена (`POST /api/v1/guest/order/{id}/cancel`) переводит бронь в терминальный
статус и освобождает слот — тем же кодом отмены, что у остальных типов.

### CMS

`SlotConfig` редактируется на позиции типа `slot`:

| Метод | Путь |
|---|---|
| GET | `/api/v1/cms/items/{item_id}/slot-config` |
| PUT | `/api/v1/cms/items/{item_id}/slot-config` |

Тело PUT — поля `SlotConfig` (duration/capacity/schedule/execution_point/…).
Правила: `duration_minutes ≥ 5`, `capacity ≥ 1`, расписание и отдел
обязательны — иначе слоты не из чего строить (`422`).

## Трекер

Бронь приезжает на доску своего отдела тем же объектом. Тело карточки — блок
`slot` (время, ресурс, гость) вместо позиций. Доска, действия, статусы, WS —
общие. Отдельного «трекера броней» нет и не должно появиться.
