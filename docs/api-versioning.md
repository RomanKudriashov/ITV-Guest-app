# Политика версионирования API

Зачем: как только выйдет нативное приложение, у пользователя останется
установленная старая версия. Ломать контракт «на месте» станет нельзя — старый
клиент не обновится сам. Версия в пути фиксирует контракт до того, как это
станет дорого.

## Где живёт версия

* **Все** прикладные маршруты — под префиксом пути: `/api/v1/...`.
* WebSocket — так же: `/ws/v1/...`.
* Версия задаётся **в одном месте** на каждой стороне:
  * бэкенд — монтирование `path("api/v1/", api.urls)` + `ApiVersionMiddleware`;
  * фронтенд — константы `API_BASE = '/api/v1'` и `WS_BASE = '/ws/v1'`
    (`frontend/src/api/client.ts`). В вызовах путь без версии — префикс
    добавляется здесь.

## Переходные алиасы (переезд без одномоментной поломки)

Безверсионные пути `/api/...` и `/ws/...` **продолжают работать** как алиасы на
v1 — чтобы существующие клиенты, вкладки и E2E не падали в момент выкатки.

* HTTP: `ApiVersionMiddleware` переписывает `/api/<x>` → `/api/v1/<x>` **до**
  резолюции URL. Один набор вьюх обслуживает оба адреса.
* На ответ по безверсионному пути вешаются заголовки:
  * `Deprecation: true` (RFC 8594);
  * `Sunset: <дата>` — когда алиас снимут (из `API_LEGACY_SUNSET`, пусто = дата
    не анонсирована);
  * `Link: </api/v1/>; rel="successor-version"`.
* На **любой** ответ `/api/*` вешается `X-API-Version` (значение из
  `API_VERSION`).

Алиасы — временные. План снятия: перевести всех клиентов на `/api/v1/`,
объявить `Sunset`, выждать окно (см. ниже), удалить `ApiVersionMiddleware` и
безверсионные `/ws/` маршруты.

## Что считается ломающим изменением

Требует **новой мажорной версии** пути (`/api/v2/`):

* удаление или переименование эндпоинта, поля ответа, значения enum;
* сужение типа/формата поля, новое **обязательное** поле запроса;
* изменение семантики существующего поля (то же имя — другой смысл);
* изменение кодов ошибок/статусов, на которые клиент полагается;
* изменение формата WS-снимка (реконсиляция ломается молча).

**Не ломающее** (допустимо в рамках v1, без новой версии):

* новый эндпоинт;
* новое **необязательное** поле запроса (с дефолтом);
* новое поле в ответе (клиент игнорирует незнакомое);
* новое значение enum, если клиент уже обязан игнорировать незнакомые;
* внутренние изменения без влияния на контракт.

## Как вводится v2

1. Поднять `/api/v2/` рядом с `/api/v1/` (обе живут параллельно).
2. Обновить контракты: в затронутых `docs/*-api-contract.md` — раздел «v2:
   что изменилось», старую форму пометить `deprecated`.
3. Перевести клиентов; на v1 повесить `Sunset`.
4. По истечении окна — снять v1.

## Сколько живут старые версии

* Безверсионные алиасы (`/api/...`) — до перевода всех клиентов на `/api/v1/`;
  на MVP-этапе (один веб-клиент) снимаются сразу после того, как фронт и E2E
  переехали.
* Мажорная версия (`v1` после выхода `v2`) живёт не меньше **срока, пока в
  проде остаются установленные приложения на ней** — ориентир — 12 месяцев с
  анонса `Sunset`. Точная дата — в `API_LEGACY_SUNSET`.

## Тесты

`test_api_versioning` держит инвариант: и `/api/v1/...`, и алиас `/api/...`
отвечают одинаково; `X-API-Version` присутствует; на алиасе есть `Deprecation`.
Ослаблять эти проверки, чтобы «прошло», нельзя — они и есть гарантия переезда.
