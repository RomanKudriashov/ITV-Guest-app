# Контракт уведомлений и эскалации

Фиксируется **до** реализации. Префикс CMS-эндпоинтов: `/api/v1/cms`.
Аутентификация и тенант — как у остальной CMS (`docs/cms-api-contract.md`).

## Зачем это

Заявка бесполезна, если её никто не увидел. Отдел получает уведомление в свой
канал; если за отведённое время заявку не взяли в работу — она поднимается
выше по ступеням: старший смены, затем руководитель. Взяли в работу —
эскалация гаснет.

## Гарантии движка

Три свойства, ради которых он вообще написан:

1. **Ступень заново проверяет состояние заказа в момент исполнения.**
   Отмена запланированной задачи — оптимизация, а не гарантия: задача может
   сработать ровно в ту секунду, когда официант жмёт «Принять». Поэтому перед
   отправкой ступень перечитывает заказ и молча гасит себя, если он уже принят,
   завершён или отменён. Это та же дисциплина реконсиляции, что у WebSocket.
2. **Идемпотентность.** У каждой отправки есть ключ дедупликации
   (`hotel + заказ + ступень + канал`) с уникальным индексом. Повтор задачи —
   обычное дело для Celery — не приводит ко второму сообщению гостю или
   сотруднику.
3. **Недоступность канала не задевает заказ.** Отправка идёт отдельной задачей
   с ретраями и экспоненциальным backoff. Упавший Telegram делает запись
   `failed` в журнале и ничего больше.

## 1. Модель

### NotificationChannel — куда слать

```jsonc
{
  "id": "...", "type": "telegram",          // telegram | email | log
  "title": "Чат кухни",
  "is_active": true,
  "execution_point_id": "..." | null,       // канал отдела
  "user_id": "..." | null,                  // персональный канал сотрудника
  "config": {"chat_id": "-100123"},         // секреты не отдаются наружу
  "config_public": {"chat_id": "-100123"},  // то же без секретов — для CMS
  "templates": {
    "ru": {"subject": "Новая заявка №{{number}}", "body": "..."},
    "en": {"subject": "...", "body": "..."}
  }
}
```

`config` по типам:

| Тип | Поля | Секретные |
|---|---|---|
| `telegram` | `bot_token`, `chat_id` | `bot_token` |
| `email` | `to` (список), `from_email` | — |
| `log` | — | — |

`log` — адаптер для разработки и CI: пишет сообщение в лог приложения и всегда
успешен. Без него тесты и демо требовали бы настоящих кредов.

**Секреты наружу не отдаются.** В ответах вместо `config` приходит
`config_public` с замаскированными значениями (`bot_token: "••••1234"`).
Записать секрет можно, прочитать — нет.

### EscalationRule + EscalationStep — когда и кому

```jsonc
{
  "id": "...", "name": "Кухня: подъём по смене",
  "execution_point_id": "...",   // null — правило по умолчанию для отеля
  "is_active": true,
  "steps": [
    {"id": "...", "sort_order": 0, "delay_minutes": 0,
     "target_kind": "point",   "channel_id": null,  "title": "Сразу — в чат кухни"},
    {"id": "...", "sort_order": 1, "delay_minutes": 5,
     "target_kind": "lead",    "channel_id": null,  "title": "Через 5 мин — старшему"},
    {"id": "...", "sort_order": 2, "delay_minutes": 15,
     "target_kind": "manager", "channel_id": null,  "title": "Через 15 мин — руководителю"}
  ]
}
```

`delay_minutes` считается **от создания заказа**, а не от предыдущей ступени:
так «через 15 минут» означает ровно то, что написано, и перенастройка одной
ступени не сдвигает остальные.

`target_kind`:

| Значение | Кому уходит |
|---|---|
| `point` | все активные каналы точки исполнения |
| `lead` | персональные каналы старших смены этой точки |
| `manager` | персональные каналы руководителей этой точки |
| `channel` | конкретный канал из `channel_id` |

Если цель не разрешилась ни в один канал, ступень пишет в журнал `skipped` с
причиной — молчаливое «никому не ушло» недопустимо.

### NotificationLog — журнал и состояние

```jsonc
{
  "id": "...", "order_id": "...", "order_number": 42,
  "rule_id": "...", "step_id": "...", "step_index": 1,
  "channel_id": "..." | null, "channel_type": "telegram",
  "target_kind": "lead",
  "status": "sent",            // scheduled | sent | failed | skipped | cancelled
  "scheduled_for": "2026-07-22T18:05:00+03:00",
  "sent_at": "2026-07-22T18:05:01+03:00",
  "attempts": 1,
  "error": "",
  "subject": "Новая заявка №42",
  "body": "Комната 305 · Такси · Аэропорт Пулково",
  "accepted_at_send": false     // был ли заказ уже принят в момент срабатывания
}
```

Записи двух уровней: у ступени есть **родительская** запись (`channel_id: null`
— «ступень сработала») и **дочерние** по одной на канал («сообщение ушло»).
Так видно и то, что ступень отработала, и куда именно она разошлась.

Статусы:

* `scheduled` — запланирована, ждёт своего времени;
* `sent` — отправлено;
* `failed` — канал не принял после всех ретраев;
* `skipped` — ступень сработала, но слать было некому/незачем;
* `cancelled` — заказ приняли раньше, ступень погашена.

## 2. Жизненный цикл

```
order.created ─→ plan_escalation(order)
                   ├─ ступень 0 (delay 0) → выполняется сразу
                   └─ ступени 1..N → Celery apply_async(countdown=delay)

каждая ступень при исполнении:
   перечитать заказ ──→ принят / завершён / отменён ? ──→ cancelled, выход
                    └─→ нет ──→ разрешить цели ──→ отправки (по задаче на канал)

order.accepted / order.status_changed(терминальный) / order.cancelled
   └─→ cancel_pending(order): scheduled → cancelled, revoke задач (best-effort)
```

`order.accepted` — новое событие шины: раньше принятие заказа было неотличимо
от любой другой смены статуса, а для эскалации это ключевой момент.

## 3. CMS API

### Каналы

| Метод | Путь |
|---|---|
| GET | `/api/v1/cms/notification-channels` |
| POST | `/api/v1/cms/notification-channels` |
| PATCH / DELETE | `/api/v1/cms/notification-channels/{id}` |
| POST | `/api/v1/cms/notification-channels/{id}/test` |

`test` отправляет пробное сообщение и возвращает результат — настраивать канал
вслепую и ждать первой настоящей заявки, чтобы узнать про опечатку в токене,
нельзя.

Валидация конфигурации по типу: `422 channel_config_invalid` с `field`.

### Правила эскалации

| Метод | Путь |
|---|---|
| GET | `/api/v1/cms/escalation-rules` |
| POST | `/api/v1/cms/escalation-rules` (вместе со ступенями) |
| PATCH | `/api/v1/cms/escalation-rules/{id}` (`steps` заменяют набор целиком) |
| DELETE | `/api/v1/cms/escalation-rules/{id}` |

Правила:
* `delay_minutes` неотрицательный, ступени упорядочены по возрастанию задержки —
  иначе `422 steps_out_of_order`;
* две ступени с одинаковой задержкой — `422 duplicate_delay`;
* `target_kind=channel` без `channel_id` — `422 channel_required`;
* правило без ступеней — `422 rule_without_steps`;
* на одну точку исполнения — не больше одного активного правила
  (`409 rule_already_exists`).

### Журнал

`GET /api/v1/cms/notification-log?order_id=&status=&limit=` — новые сверху.

## 4. Настройки

| Переменная | Смысл |
|---|---|
| `NOTIFICATIONS_ENABLED` | глобальный выключатель (в тестах `0`) |
| `NOTIFICATIONS_FALLBACK_ADAPTER` | чем слать, если у канала нет кредов (`log`) |
| `TELEGRAM_API_URL` | база API бота — подменяется в тестах |
| `DEFAULT_FROM_EMAIL` | отправитель писем |

Задержки ступеней задаются данными (`delay_minutes`), а не настройками: в
тестах ступени вызываются напрямую, поэтому ждать реальных минут не нужно
никогда.
