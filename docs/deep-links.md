# Deep-links: канонический вид ссылки входа гостя

Зачем фиксировать сейчас: ссылка `/r/<номер>` зашита в **печатные QR-коды**,
которые уже клеятся в номерах. Сменить схему задним числом = перепечатать QR по
всему отелю. Значит форму нужно зафиксировать с запасом под приложение, ТВ-QR и
ключи со стойки — до того, как QR разъедутся по номерам.

## Канонический вид

```
<адрес отеля>/r/<номер>[?lang=<код>&src=<источник>&t=<разовый токен>]
```

* `<адрес отеля>` — `Hotel.public_guest_url()`: кастомный домен отеля, иначе
  поддомен на базовом домене платформы.
* `/r/<номер>` — путь входа гостя. Префикс `/r/` — **единственная константа**
  схемы (`Hotel.DEEPLINK_ROOM_PREFIX`); менять нельзя.
* Query-параметры — **задел на будущее**, сейчас необязательны и поведение
  витрины не меняют:
  * `lang` — предвыбор языка (приложение/ТВ смогут открыть сразу на языке гостя);
  * `src` — источник входа (`qr`, `tv`, `frontdesk`, `applink`…) для аналитики;
  * `t` — разовый токен (ключ со стойки/PMS-подтверждение) — место под
    доверенный вход без ручного ввода номера.

Параметры именованы и опциональны, поэтому старые QR (без параметров) остаются
валидными вечно, а новые сценарии добавляются, не ломая печатные коды.

## Одна функция формирования

Deep-link собирается **только** через `Hotel.room_deeplink(room_number, *,
lang=None, source=None, token=None)`. И генерация QR, и матрица номеров, и любой
будущий клиент зовут её — чтобы схема не разъехалась склейкой строк в разных
местах (ровно то, из-за чего схемы потом расходятся). `public_guest_url()`
остаётся низкоуровневой (домен + путь), а знание про `/r/` живёт в одном методе.

## Поведение страницы `/r/<номер>`

* В браузере — как раньше (прогон 3): открывается витрина, гость входит по
  номеру. Экран и логика не меняются.
* Несуществующий номер — аккуратный фолбэк на ручной ввод (`room_not_found`,
  `hint: manual_entry`). Ссылка остаётся валидной, тупика нет.
* При установленном приложении ОС сама откроет его (universal/app links) —
  **без** «выберите, где открыть» и **без** баннеров «поставьте приложение».

## Готовность к universal links (iOS) и app links (Android)

Файлы связи сайта с приложением отдаются по стандартным путям в корне домена:

| Путь | Content-Type | Примечание |
|---|---|---|
| `/.well-known/apple-app-site-association` | `application/json` | **без расширения** (требование Apple) |
| `/.well-known/assetlinks.json` | `application/json` | Android |

* Отдача **выключена конфигом** `APP_LINKS_ENABLED` (по умолчанию `False`): пока
  приложения нет, эндпоинты возвращают `404`, чтобы ОС не пыталась открыть
  несуществующее приложение. Появится приложение — один переключатель включает
  ассоциации, печатные QR начинают открываться в приложении **без перепечатки**.
* Идентификаторы (`IOS_APP_ID`, `ANDROID_PACKAGE`,
  `ANDROID_SHA256_FINGERPRINTS`) — **плейсхолдеры**, не реальные. Заполнить
  перед включением (граница этого прогона — реальные id не заводим).
* Пути ассоциации указывают на `/r/*` — приложению делегируются только
  deep-links входа, не весь сайт.

## Что НЕ делаем сейчас

Само приложение, реальные идентификаторы в файлах связи, ТВ-поверхность, выдачу
разовых токенов со стойки. Здесь только **схема и готовность** — место под них
заложено, реализация — отдельные прогоны.
