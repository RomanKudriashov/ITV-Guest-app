# ITV Guest App

Мультиотельная гостевая платформа: гость сканирует QR в номере, со своего
телефона смотрит меню и услуги отеля и заказывает; персонал ведёт заявки в
трекере; отель управляет каталогом и оформлением.

Один движок на все отели — отличаются только данные и тумблеры.

> **Что умеет.** Гость заказывает еду корзиной или оформляет
> заявку-услугу формой → заявка приходит на доску своего отдела и **уведомляет
> его по каналу**; если за X минут не приняли — поднимается выше по ступеням
> (старший → руководитель); приняли — эскалация гаснет. Один `Order`, один
> трекер, один гостевой поток на все четыре типа. Отель видит всё в **аналитике**:
> продажи, операции, трафик, отзывы — по предагрегатам, собранным из событий.

### Как попасть в конкретный отель

Отель определяется **поддоменом**: `crystal.guest.localhost`. Локально это
работает без правки `/etc/hosts` — всё, что оканчивается на `.localhost`,
резолвится в 127.0.0.1 (macOS, Linux, современные браузеры). Через curl —
заголовком `Host`:

```bash
curl -H "Host: crystal.guest.localhost" http://localhost:8010/api/guest/menu
```

В деве (`DJANGO_DEBUG=1`) дополнительно принимаются `X-Hotel-Subdomain: crystal`
и `?hotel=crystal`. В проде оба отключены.

### Дымовой сценарий целиком

```bash
H="Host: crystal.guest.localhost"
BASE=http://localhost:8010/api/guest

# 1. Сессия по номеру комнаты
TOKEN=$(curl -s -X POST -H "$H" -H "Content-Type: application/json" \
  -d '{"room_number":"305"}' $BASE/session | jq -r .token)

# 2. Меню — локализованное, с учётом расписания
curl -s -H "$H" -H "Authorization: Bearer $TOKEN" -H "Accept-Language: en" $BASE/menu | jq

# 3. Заказ — идемпотентный
curl -s -X POST -H "$H" -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" -H "Idempotency-Key: demo-1" \
  -d '{"lines":[{"item_id":"<id>","quantity":2,"modifier_option_ids":["<id>"]}]}' \
  $BASE/order | jq

# 4. Статус
curl -s -H "$H" -H "Authorization: Bearer $TOKEN" $BASE/order/<order_id> | jq
```

## Что уже стоит в фундаменте

### 1. Мультитенантность — эшелонированная

Три независимых рубежа, потому что цена ошибки — данные чужого отеля.

**Резолюция.** `TenantMiddleware` достаёт поддомен из `Host`, находит отель и
кладёт его в контекст запроса (`apps/core/context.py`) — и в питон-contextvar,
и в сессионную переменную Postgres `app.current_hotel`.

**ORM.** `TenantModel.objects` — это `TenantManager`, который сам добавляет
фильтр по текущему отелю. Прикладной код фильтр по `hotel_id` **не пишет**.
Поведение fail-closed: вне контекста отеля запрос вернёт пусто, а не чужое.

**Postgres RLS.** На каждой тенант-таблице политика
`hotel_id::text = current_setting('app.current_hotel', true)` плюс
`FORCE ROW LEVEL SECURITY`. Это страховка от того, что пройдёт мимо менеджера:
raw SQL, `.all_objects`, невнимательный related-запрос.

Три роли БД (`infra/postgres/init/01-roles.sh`):

| Роль | Назначение |
|---|---|
| `postgres` | bootstrap-суперпользователь образа. Только инициализация. **Приложение под ним не работает — суперпользователь игнорирует RLS.** |
| `guestapp` | роль приложения. Обычная, без BYPASSRLS. Под ней backend и worker. |
| `guestapp_platform` | платформенная, BYPASSRLS. Миграции и кросс-отельный уровень. |

Поэтому миграции гоняются платформенной ролью:

```bash
docker compose run --rm --entrypoint python backend manage.py migrate --database=platform
```

Выйти за пределы одного отеля можно только осознанно: `platform_scope()` снимает
фильтр ORM, но RLS остаётся — читать кросс-отельно получится лишь через
`.using("platform")`. Двойной замок сделан нарочно: случайно так не напишешь.

Новая тенант-таблица → имя в `apps/core/tenant_tables.py` + миграция с
`rls.enable_sql([...])`. Забыть нельзя: `test_rls_coverage.py` находит все
модели с `hotel` и падает, если таблицы нет в списке.

### 2. Базовый класс моделей

`BaseModel`: UUID-pk, `created_at/updated_at/created_by`, soft-delete
(в том числе при массовом `.delete()`), помощник переводимых полей.
`TenantModel` добавляет `hotel_id` и автоскоуп.

Переводимые поля — единая схема, `TranslatableField` (JSONB `{lang: value}`):

```python
item.title          # {"ru": "Стейк рибай", "en": "Ribeye steak"}  — для CMS
item.title_i18n     # строка на языке текущего запроса
item.tr("title", "ar")
```

Фолбэк: запрошенный язык → язык отеля → `en` → любое непустое. Недопереведённый
контент не роняет выдачу меню гостю.

`created_by` — UUID, а не FK: иначе core зависел бы от accounts, а accounts от
hotels, и получился бы цикл миграций. Жёсткая трассировка действий — в `AuditLog`.

### 3. Аутентификация

| Субъект | Механизм |
|---|---|
| Персонал | JWT (HS256), stateless, с `hotel` и списком точек исполнения |
| Гость | непрозрачный отзываемый токен; в БД только SHA-256 |
| Платформа | JWT со `scope=platform`, чтение через платформенное подключение |

У гостевой сессии есть **уровень доверия** (`trust`):
`anonymous < room_scanned < pms_verified < staff_verified`. Права зависят от
него, а не от факта наличия токена: смотреть меню можно анонимно, писать на
счёт номера — только подтверждённому по PMS.

Аутентификация никогда не выбирает тенанта: тенант уже выбран поддоменом,
токен обязан ему соответствовать.

Impersonation заложен каркасно: грант в БД, клейм `imp` в токене, запись в
аудит. Действие поддержки от имени сотрудника остаётся отличимым от действия
самого сотрудника.

### 4. Событийная шина и realtime

`apps/events/bus.py`. События: `order.created`, `order.status_changed`,
`order.cancelled`. Подписчики (каркас): канал трекера по WebSocket, счётчики
аналитики в Redis, аудит.

Два правила, ради которых шина и написана:

1. **Эмит после коммита** (`transaction.on_commit`). Иначе подписчик увидит
   заказ, которого в базе ещё нет, — а при откате не будет никогда. То же
   правило распространено на диспатч Celery-задач.
2. **Падение подписчика не роняет операцию.** Заказ создан; несработавший
   счётчик — не повод отдавать гостю 500.

WebSocket: `/ws/tracker/` (доска точки исполнения) и `/ws/order/<id>/` (гость
следит за своим заказом).

### 5. Идемпотентность и транзакции

`POST /api/guest/order` требует заголовок `Idempotency-Key`. Повтор с тем же
ключом и тем же телом отдаёт **тот же** заказ (200 вместо 201); тот же ключ с
другим телом — 409. Гонка одновременных запросов разрешается уникальным
индексом в БД, а не блокировкой в питоне.

Заказ, позиции и резолв маршрута создаются в одной транзакции; событие уходит
после коммита.

### 6. Медиапайплайн

Загрузка → оригинал в MinIO → Celery режет варианты (`thumb`/`card`/`full`,
WebP, с учётом EXIF-поворота) → фолбэк на заглушку по категории. API никогда не
ждёт обработки: ассет отдаётся в статусе `pending`.

### 7. Фронтенд-каркас

Тема MUI собирается **только из токенов бренда** (`frontend/src/theme/tokens.ts`),
светлая/тёмная, RTL через `stylis-plugin-rtl` с переключением emotion-cache.
i18n: ru/en/ar/zh, автоопределение языка устройства, фолбэк `en`.

**Правило проекта: ни одного захардкоженного цвета вне `tokens.ts`.** Формат
токенов совпадает с `BrandTheme.tokens` на бэкенде — это один контракт, а не
две похожие структуры.

## Админка отеля

Разделы управления в CMS: номера с QR, локации, отделы, персонал. Контракт —
[`docs/hotel-admin-api-contract.md`](docs/hotel-admin-api-contract.md).

```
        /api/cms/rooms                CRUD + bulk (диапазон)
GET     /api/cms/rooms/{id}/qr.svg|png   QR номера
GET     /api/cms/rooms/qr-sheet       печатный лист всех QR
        /api/cms/locations            CRUD + расписание
GET/PUT /api/cms/locations/matrix     матрица «категория → локации»
        /api/cms/departments          CRUD отделов (+ счётчики связей с каналами)
        /api/cms/staff                CRUD персонала + привязки к отделам
```

Решения:

* **QR кодирует рабочий deep-link** `/r/{номер}` на публичном адресе отеля
  (`Hotel.public_guest_url`) — ровно тот, что понимает гостевой вход. Скан
  открывает витрину, привязанную к номеру, без лишних экранов.
* **Добавление диапазоном идемпотентно**: существующие номера уходят в
  `skipped`, а не роняют операцию — заводить отель по частям это норма.
* **Список персонала для каналов**: `GET /api/cms/staff` даёт список сотрудников,
  и персональный канал уведомлений настраивается (выбор сотрудника).
* **Себя нельзя удалить или выключить** — иначе единственный админ запрёт себя
  снаружи. Действующий пользователь передаётся во вьюху явно: в CMS-запросе он
  не лежит в контексте актора.
* **Привязка к чужому отделу невозможна** — RLS не отдаёт точку другого отеля,
  и попытка возвращает «отдел не найден», не раскрывая факт его существования.
* **Отдел с заказами не удаляется** (заказы ссылаются через `PROTECT`) —
  только выключается.

## Бренд-настройки

Отель крутит токены темы, а рядом — **живое превью реальной гостевой витрины**,
перекрашивающееся по черновым токенам до сохранения. Контракт —
[`docs/brand-api-contract.md`](docs/brand-api-contract.md), экран —
`/cms/brand`.

```
GET   /api/cms/brand                  текущая тема отеля
PATCH /api/cms/brand                  частичные токены (deep-merge на сервере)
GET   /api/cms/brand/presets          библиотека цельных наборов
POST  /api/cms/brand/apply-preset     заменить токены пресетом целиком
GET   /api/cms/brand/fonts            курируемый список шрифтов
GET   /api/cms/brand/abstractions     фоны-подложки (data-URI)
```

Ключевое: превью и рантайм используют **один и тот же** `createAppTheme` из
токенов — не макет, а те же компоненты, что увидит гость. Сохранённые токены
уже доезжают до витрины: гостевая сессия и меню отдают `hotel.theme` из
`BrandTheme`.

Решения:

* **Пресеты собираются из опорных цветов** (`apps/hotels/brand_palette.py`):
  пресет задаёт 4–5 seed-цветов, а 18 токенов палитры на режим выводятся
  расчётом. Писать их руками значило бы плодить рассогласованные оттенки.
* **PATCH — deep-merge, apply-preset — замена целиком.** Разный смысл: «тронуть
  токен» против «начать с чистого набора». Ручная правка ставит
  `preset: custom`; применение пресета сохраняет уже загруженные логотипы.
* **Шрифты — только из курируемого списка** (лицензии и стабильность рендера),
  `fontFamily` и `headingFontFamily` валидируются по нему.
* **Фоны-абстракции — data-URI**, а не файлы из `/static`: в браузере `/static`
  уходит на vite-dev-сервер, который его не раздаёт. data-URI работает и в
  превью-фрейме, и в проде.
* **Затемнение фоновой картинки — токеном** `background.dim`, а не запечено в
  файл: одна картинка годится и для светлой, и для тёмной темы.

Демо: Crystal стартует на пресете «Вечерний консьерж», Aurora — на
«Тиффани-ночь», чтобы изоляция бренда была видна глазами.

## Гостевой контур: главная, чат, отзывы

Завершает гостевую поверхность. Контракт —
[`docs/guest-surface-api-contract.md`](docs/guest-surface-api-contract.md).

```
GET  /api/guest/home                 секции отеля, собранные из данных
GET/POST /api/guest/chat             тред гостя + отправка
WS   /ws/guest/chat/                 чат гостя, реконсиляция снимком
GET/POST /api/tracker/chat/threads[/{id}]   треды персонала
WS   /ws/staff/chat/{thread_id}/     чат персонала
GET/POST /api/guest/order/{id}/review       отзыв (один на заявку)
GET  /api/cms/reviews                отзывы отеля (приватные)
GET/PATCH /api/cms/review-settings   сбор отзывов + порог уведомления
```

Решения:

* **Главная собирается из данных**: секция появляется только при наличии
  активных категорий своего типа, порядок — из `sort_order`. Тип связан с
  маршрутом витрины в одном месте; сам набор — из БД.
* **Чат живёт при номере, а не при заявке**: гость пишет «когда завтрак?» вне
  привязки к заказу, переписка переживает оформление и закрытие заявок.
* **Чат — та же реконсиляция снимком, что у заказа**: WS шлёт полный снимок
  треда при подключении и на каждое сообщение любой стороны; `mine`
  вычисляется по стороне запроса. У WS нет middleware, поэтому авторизация
  явная: гость ↔ только свой тред, сотрудник ↔ треды своего отеля (RLS).
* **Отзывы приватные по умолчанию**: оценка уходит отелю, гостям не
  публикуется — сознательно против антипаттерна «витрина отзывов». Один отзыв
  на заявку. Низкая оценка (`≤ порога`) эмитит событие → уведомление менеджеру
  через существующие каналы уведомлений (service recovery до отъезда гостя).
* **Уведомления о чате и отзывах — через существующие каналы**, без
  новой инфраструктуры.

## Аналитика отеля

Контракт —
[`docs/analytics-api-contract.md`](docs/analytics-api-contract.md).

**Главное решение — не считать на лету по живым заказам** (это боль ITV), а
копить предагрегаты через событийную шину. Дашборд читает готовые роллапы.

```
apps/analytics/collector.py   record() → журнал + редьюсер apply()
apps/analytics/models.py      AnalyticsEvent (сырой журнал) + *Daily (роллапы)
apps/analytics/queries.py     summary/timeseries/breakdown/operations/traffic/reviews/drilldown
apps/analytics/recompute.py   пересчёт из журнала (и восстановление из заказов)
GET /api/cms/analytics/*      дашборд; POST .../export → CSV/XLSX фоном (Celery)
```

Устройство сбора:

* Каждое доменное событие (`order.*`, `review.created`, `session.started`)
  проходит через **один вход** `record(...)`: пишет сырой факт в журнал
  (дедуп по натуральному ключу) и — только для нового факта — инкрементит
  дневные роллапы **редьюсером**, который читает лишь слепок сырой строки.
* **Идемпотентность** — из дедупа: повтор события не двоит счётчик.
* **Пересчёт == живая агрегация** — by construction: `recompute` прогоняет тот
  же редьюсер по тому же журналу, поэтому даёт те же числа. Есть и
  восстановление журнала из живых заказов (для истории и починки расхождений).
* Все группировки по времени — в **сутках отеля** (`hotel.tzinfo`), не UTC.
* Скоуп прав — теми же привязками, что трекер: супер-админ по всем отелям,
  админ отеля — весь отель, **старший точки — только свою точку**; тенант-
  изоляция сверх этого — RLS.
* Демо-история — под флагом `--with-analytics-history` (по образцу
  `--with-guest-history`), чтобы не сдвигать нумерацию заказов в тестах.

## Бронь слотов (тип `slot`)

Гость выбирает дату → видит свободные слоты (рабочие часы минус занятые с
учётом вместимости) → бронирует. Контракт —
[`docs/slot-info-api-contract.md`](docs/slot-info-api-contract.md).

**Защита от двойной брони** — ядро типа. Бронь идёт в транзакции с
`SELECT … FOR UPDATE` на строке `SlotConfig`: это сериализует одновременные
брони одного ресурса. Проверка вместимости внутри блокировки не может быть
обогнана — два гостя за последний слот: успевает один, второй получает
`slot_taken`. Есть тест на реальной гонке (два потока, настоящие транзакции):
из двух одновременных броней слота с вместимостью 1 проходит ровно одна.

Отмена (гостем или персоналом) снимает `SlotBooking.is_active` — и слот
освобождается. Живёт в общей смене статуса, поэтому работает для обоих.

```
GET  /api/guest/slots?item_id=&date=   свободные слоты на дату
POST /api/guest/order {slot_start}     бронь тем же эндпоинтом
GET/PUT /api/cms/items/{id}/slot-config  настройка ресурса
```

## Уведомления и эскалация

Заявка бесполезна, если её никто не увидел. Отдел получает сообщение в свой
канал; не приняли за отведённое время — заявка поднимается выше по ступеням.
Контракт — [`docs/notifications-api-contract.md`](docs/notifications-api-contract.md),
настройка — `/cms/notifications`.

```
GET/POST/PATCH/DELETE  /api/cms/notification-channels[/{id}]
POST                   /api/cms/notification-channels/{id}/test
GET/POST/PATCH/DELETE  /api/cms/escalation-rules[/{id}]
GET                    /api/cms/notification-log?order_id=&status=
```

Каналы — за интерфейсом-адаптером (`apps/notifications/channels/`): Telegram,
e-mail и **log** — рабочий канал для разработки и CI, чтобы стенд не требовал
бота и SMTP. Новый канал (WhatsApp, SMS) = новый адаптер, движок не трогаем.

Три гарантии движка, ради которых он написан:

* **Ступень заново проверяет состояние заказа при исполнении.** Отмена
  запланированной задачи — оптимизация, а не гарантия: задача может сработать
  ровно в секунду, когда официант жмёт «Принять». Поэтому перед отправкой
  ступень перечитывает заказ и гасит себя, если он уже принят. Та же дисциплина
  реконсиляции, что у WebSocket.
* **Идемпотентность.** У каждой отправки — ключ дедупликации с уникальным
  индексом; повтор Celery-задачи не даёт второго сообщения.
* **Недоступность канала не задевает заказ.** Отправка — отдельная задача с
  ретраями и backoff; упавший Telegram оставляет `failed` в журнале и ничего
  больше. Неповторяемая ошибка (неверный токен) не ретраится впустую.

Секреты каналов (`bot_token`) можно записать, но нельзя прочитать: наружу
уходит маска `••••1234`. `delay_minutes` считается от создания заказа, а не от
предыдущей ступени. Тайминги — данные, не настройки: в тестах ступени
вызываются напрямую, ждать реальных минут не нужно.

Демо: у кухни канал «Чат кухни» (log) и правило `сразу → +5 мин старшему →
+15 мин руководителю`.

## Типы предложений

Система обслуживает разные виды услуг **одним кодом**. Различия собраны в одну
таблицу поведений (`apps/catalog/offerings.py`), а не разбросаны условиями —
подробности и правило для будущих типов в
[`docs/offering-types.md`](docs/offering-types.md).

| | `product` | `service_request` | `info` | `slot` |
|---|---|---|---|---|
| Чем наполняется | корзина | форма полей | — (чтение) | выбор слота |
| Создаёт заказ | да | да | **нет** | да (`booking`) |
| Тело карточки | позиции | `field_values` | `content` | `slot` |
| Отдел | кухня | консьерж/хозслужба | — | SPA |

`info` — единственный тип без заказа: страница только для чтения (Wi-Fi, об
отеле). `slot` — бронь ресурса: гость выбирает дату → свободные слоты →
бронирует → обычный заказ типа `booking`, дальше тот же поток.

Общее у типов: `Order`, маршрутизация, статусы, события, WebSocket,
идемпотентность, история, трекер, изоляция тенантов. Ни одной ветки по типу.

Заявка — это заказ с **одной** позицией и заполненным `field_values`. Позиция
у неё есть намеренно: на ней держатся маршрут, снапшот названия, проверка
доступности и расчёт суммы. Убрать её значило бы завести второй путь для всего
перечисленного.

```
GET  /api/guest/catalog?type=service_request   каталог услуг (тот же эндпоинт)
GET  /api/guest/item/{id}                      у заявки — request_fields
POST /api/guest/order                          + field_values, тот же эндпоинт
     /api/cms/items/{id}/request-fields        конструктор полей в CMS
```

Демо-услуги: **Такси** → консьерж (`concierge@crystal.local`), **Уборка** →
хозслужба (`maid@crystal.local`). Пароль у всех — `chef12345`.

### Единое правило состояния

Обязательное правило к переиспользованию: без него редакторы CMS ведут
себя по-разному, и оба варианта неверны — один навсегда застревает в
«не сохранено», другой может затереть правки.

1. Серверные данные — только через react-query.
2. **Незавершённый ввод пользователя** (корзина, форма заказа, выбранные
   модификаторы) живёт в отдельном черновике `useDraftState` и **никогда** не
   перезаписывается фоновым refetch. Пересев черновика — только при смене
   сущности, к которой он привязан.
3. **Живой статус — реконсиляция:** пришёл снимок по WS →
   `queryClient.setQueryData(key, snapshot)`. Никаких «накатов» дельт.
4. Корзина привязана к `session_id`, переживает перезагрузку и чистится после
   успешного оформления.

Решения, которые стоит знать:

* **Расписание считает сервер, и только в таймзоне отеля.** «Кухня до 23:00» —
  это 23:00 у отеля, а не на сервере и не в телефоне гостя. Поэтому в меню
  приходит не только флаг «недоступно», но и `available_from` («с 07:00»).
* **Один расчёт доступности на всю систему** (`apps/catalog/availability.py`) —
  и для показа меню, и для валидации заказа. Расхождение здесь дало бы худший
  баг витрины: позицию, которую видно, но нельзя заказать.
* **Живой статус — реконсиляция, а не дельты.** WebSocket шлёт ПОЛНЫЙ снимок
  заказа сразу после подключения и на каждое изменение; клиент своё состояние
  заменяет, а не докручивает. Это снимает целый класс багов рассинхрона —
  пропущенное сообщение, переподключение после потери сети, гонка WS с REST.
  Формат снимка буквально тот же, что у REST.
* **Модификаторы живут в карточке блюда, а не в списке меню.** В списке — лишь
  признак `has_required_modifiers`: витрине этого хватает, чтобы решить,
  открывать карточку или добавлять позицию в один тап.
* **Отмена гостем настраивается отелем** (`allows_guest_cancel` в пресете
  статусов): «Новый» и «Принят» — да, «Готовится» — уже нет. Проверка на
  сервере, даже когда кнопки в UI нет: между отрисовкой экрана и нажатием
  кухня успевает взять заказ в работу.
* **Уровень доверия ограничивает действия, а не просмотр.** Сессия без номера
  (`anonymous`) меню видит, но заказать не может — доставлять некуда.
* **Неизвестный номер — развилка сценария, а не ошибка.** Ответ `404` несёт
  бренд отеля и подсказку `manual_entry`, чтобы гость не упёрся в голую
  системную страницу.

## Трекер кухни

Доска заказов точки исполнения для персонала. Контракт —
[`docs/tracker-api-contract.md`](docs/tracker-api-contract.md).

Открыть: http://localhost:5183/tracker (вход `chef@crystal.local` / `chef12345`).

```
GET  /api/tracker/points               точки, к которым привязан сотрудник
GET  /api/tracker/orders?point=&scope= доска: колонки из пресета статусов отеля
GET  /api/tracker/order/{id}
POST /api/tracker/order/{id}/accept    взять в работу
POST /api/tracker/order/{id}/status    двинуть статус
POST /api/tracker/order/{id}/cancel
WS   /ws/tracker/{point_code}?token=…  доска в реальном времени
```

**У WebSocket нет HTTP-middleware** — ни аутентификации, ни резолвера тенанта,
ни языка. Консьюмер обязан делать всё это сам и явно, иначе канал окажется
открыт наружу. Поэтому проверки живут в `apps/orders/tracker.py` и вызываются
и из REST, и из WS — разъехаться им негде. Проверок три: токен, отель токена
и **привязка сотрудника к точке** (`StaffAssignment`). Повар с кухни не видит
заявок бара, сотрудник соседнего отеля — вообще ничего.

Ещё несколько решений:

* **Колонки доски строит сервер** из пресета статусов отеля, как и
  `next_statuses` для кнопок карточки. Клиент не знает правил переходов —
  иначе кнопки рано или поздно разойдутся с тем, что сервер примет.
* **Полный снимок доски на каждое событие**, как и у гостя. Размен осознанный:
  на кухне одновременно живут единицы заказов, и простота инварианта важнее
  экономии трафика.
* **Повторное «Принять» отвечает 409** с именем того, кто уже принял. Два
  официанта, нажавшие одновременно, — обычное дело, и молчаливый перехват был
  бы неприятным сюрпризом для того, кто уже понёс заказ.
* **Маршрут заказа резолвится с фолбэками**: явный `Route` → точка с тем же
  кодом, что у категории → единственная точка отеля. Категория, только что
  созданная в CMS, маршрута ещё не имеет, а заказать её гость уже может —
  ронять заказ из-за ненастроенной админки нельзя.

## CMS «Меню»

Раздел, которым отель наполняет каталог. Точный контракт —
[`docs/cms-api-contract.md`](docs/cms-api-contract.md), интерактивно —
`/api/docs`.

Открыть: http://localhost:5183/login, вход в демо-отеле
**`chef@crystal.local` / `chef12345`** (у второго отеля — `chef@aurora.local`).

Экраны:

* **Меню** — дерево категорий с drag-and-drop сортировкой, счётчиками блюд и
  тумблерами; справа список блюд выбранной категории с миниатюрами, ценой,
  флагами и переключателями «активно» / «в наличии».
* **Редактор блюда** — названия и описания по табам языков (видно, где перевод
  заполнен), цена, фото с загрузкой и превью, флаги и аллергены чипами, группы
  модификаторов с опциями и правилами выбора, расписание и day-parting,
  стоп-лист. Живая валидация, индикатор несохранённых изменений.
* **Редактор категории** — названия по языкам, картинка, родитель, порядок,
  активность, расписание.

Фронт всегда шлёт заголовок `X-Hotel-Subdomain` (в деве поддомена нет: браузер
ходит на `localhost:5183` через vite-прокси). В проде тенант берётся из Host,
а заголовок игнорируется.

```
POST   /api/staff/auth/login          вход, JWT
GET    /api/staff/auth/me

GET    /api/cms/bootstrap             языки, флаги, аллергены, расписания, валюта

GET    /api/cms/categories            дерево категорий
POST   /api/cms/categories
PATCH  /api/cms/categories/{id}
DELETE /api/cms/categories/{id}?cascade=
POST   /api/cms/categories/reorder
POST   /api/cms/categories/{id}/toggle

GET    /api/cms/items?category_id=&search=
POST   /api/cms/items
GET    /api/cms/items/{id}            блюдо вместе с модификаторами
PATCH  /api/cms/items/{id}
DELETE /api/cms/items/{id}
POST   /api/cms/items/reorder
POST   /api/cms/items/{id}/stock      стоп-лист
POST   /api/cms/items/{id}/toggle
PUT    /api/cms/items/{id}/images     набор и порядок фото

POST   /api/cms/items/{id}/modifier-groups
PATCH  /api/cms/modifier-groups/{id}

POST   /api/cms/items/{id}/modifier-groups/reorder
POST   /api/cms/modifier-groups/{id}/options
PATCH  /api/cms/modifier-options/{id}

POST   /api/cms/media                 multipart, оригинал → MinIO, варианты → Celery
GET    /api/cms/media/{id}            статус обработки

POST   /api/cms/schedules
PATCH  /api/cms/schedules/{id}

```

Весь `/api/cms` закрыт JWT персонала **на уровне роутера** — забыть `auth` на
отдельном эндпоинте физически невозможно.

Несколько решений, которые стоит знать:

* **Правила модификаторов проверяет сервер.** `single` ⇒ `max_choices=1`,
  `required` ⇒ `min_choices≥1`, минимум не больше числа активных вариантов,
  обязательная группа не может остаться пустой. От этих правил зависит, что
  получит кухня, — полагаться на дисциплину формы нельзя.
* **Коды генерируются с транслитерацией.** «Горячее» → `hot`… точнее,
  `goryachee`: `slugify` выбрасывает кириллицу целиком, и код схлопнулся бы в
  пустоту. Коды попадают в URL и интеграции, поэтому только латиница.
* **Флаги и аллергены — общий справочник** (`apps/catalog/vocabularies.py`),
  а не свободный текст: иначе «vegan» и «vegetarian» разъедутся по отелям и
  фильтрация в витрине перестанет работать. Сервер отвергает незнакомый код.
* **Статические пути объявляются раньше параметризованных.** `/items/reorder`
  обязан идти до `/items/{id}` — Django резолвит URL по порядку регистрации.
* **Удаление категории с блюдами** возвращает `409` с числом блюд и требует
  явного `cascade=true`. Блюда удаляются мягко: история заказов ссылается на
  них через `PROTECT` и обязана пережить удаление позиции из меню.

## Модель данных (срез «еда»)

`Hotel` · `BrandTheme` · `HotelLanguage` · `Room` · `ExecutionPoint` ·
`Location` · `Schedule`+`ScheduleInterval` — отель и его структура.
`User`+`StaffAssignment` · `GuestSession` — доступ.
`Category` (дерево) · `Item` · `ModifierGroup`+`ModifierOption` · `Route` ·
`ServiceLocation` — каталог.
`Order`+`OrderItem` · `StatusDefinition` · `OrderStatusChange` — заказы.
`MediaAsset` · `CategoryPlaceholder` · `AuditLog` · `IdempotencyKey`.

Два сквозных решения:

* **Цены — в минимальных единицах** (копейках), целыми. Никакого float и
  никаких сюрпризов округления.
* **Время — в UTC, расписания — в таймзоне отеля.** «Кухня до 23:00» означает
  23:00 у отеля, а не на сервере.

`Category` и `Item` уже имеют поле `type`. Еда — это `type=product`; SPA,
экскурсии и трансфер лягут в те же таблицы, а не в параллельную иерархию.

Заказ хранит **снапшоты**: цену, название и модификаторы на момент заказа, и
резолвленную точку исполнения. Меню меняется каждый день — история заказа
меняться не должна.
